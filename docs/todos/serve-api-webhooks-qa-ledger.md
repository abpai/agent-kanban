**OUT OF SCOPE** (will not touch): provider internals `src/providers/*` including the bodies of `provider.handleWebhook` (jira/linear event parsing); CLI command dispatch (`board/task/column/bulk/comment/config` routing in `index.ts`); **`parseMcpArgs` and the `mcp` entry branch** in `index.ts`; the MCP layer `src/mcp/*`; the UI `ui/*`; `db.ts` internals; storage/sync/tracker config modules. The API router exercises the local provider + `db.ts` only as a black-box dependency.

**Boundary notes:**

- `parseEntryArgs` is IN scope **only** as the error-envelope wrapper for `parseServeArgs`; its use around `parseMcpArgs` is out.
- `webhooks.ts` HMAC helpers are shared security primitives also consumed by providers; testing here targets the primitives + the API-level webhook _route_ (target match, unsupported-provider, unauthorized passthrough, handled/mutated tagging), NOT provider-internal event semantics.

## Non-goals

- No new endpoints/features — this is QA, not feature work.
- No changes to provider-internal webhook event parsing.
- No CLI / MCP / UI changes.
- No dependency upgrades.
- No changes to the public `{ ok, data | error }` JSON envelope contract without an explicit, Reviewer-approved defect justification.

## Severity rubric

| Severity     | Definition (this scope)                                                                                                                                                                              |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Critical** | Auth bypass, signature-verification bypass, path traversal / arbitrary file read, RCE, data loss, or a crash that takes the server down on a reachable input.                                        |
| **High**     | Wrong status/envelope on a security-relevant path, a mutation that broadcasts/persists incorrectly, an unhandled rejection on a reachable request, or a resource/timer leak across server lifecycle. |
| **Medium**   | Incorrect-but-contained behaviour (wrong non-security status code, mis-tagged `mutated`, sloppy validation) with a clear correct answer and no security impact.                                      |
| **Low**      | Cosmetic, log-noise, minor spec drift, or hardening nice-to-haves.                                                                                                                                   |

**Defect resolution policy:** every defect is either (a) **fixed** — smallest safe fix, one commit, verified locally — or (b) **accepted** — only with an explicit written rationale AND Reviewer sign-off recorded in the Reviewer gate log. **No critical/high defect may be accepted** to satisfy an exit criterion; it must be fixed or the loop STOPs (Blocked). Severity is set by the Executor and may be challenged/raised by the Reviewer; the higher of the two stands until evidence resolves it.

### Test-category matrix (for EC2)

Categories considered per feature: **Happy · Error · Boundary · InvalidInput · Security · Perf · Responsive**.

- **Responsive** is **N/A for the entire scope** (no UI in scope) — record once here, not per row.
- **Perf** is N/A for pure request/response logic; it is in-scope only where the code makes a timing/throughput claim (F11 background-sync interval, F14 idleTimeout) and is satisfied by an assertion on the scheduling/timeout behaviour, not a benchmark.
- A feature may legitimately have categories that don't apply (e.g. a read route has no InvalidInput body) — those are marked `N/A — <reason>` in the Test Cases cell.

### Discovery-pass procedure (for EC7)

One discovery pass =

1. `grep`/AST-scan the in-scope files for every exported symbol, route literal, `req.method`/path branch, and error path; diff that set against the ledger feature rows.
2. Re-read each in-scope file end-to-end for branches with no test (use `bun test --coverage` on the in-scope files to find unexecuted lines).
3. Adversarially probe for new defects on the surfaces touched since the last pass.

A pass "finds zero new features" iff step 1–2 add no new feature row; "zero new defects" iff step 3 opens no new Defects row. Two consecutive zero/zero passes → Done (in combination with EC1–EC8).

## Circuit breakers (STOP + hand back)

- **Done** — **EC1 through EC8 all satisfied** (EC7's two-zero-passes is necessary but NOT sufficient; tests, defects, full check, and final Reviewer approval must all hold).
- **Blocked** — needs human input / credentials / a decision not derivable from code+docs (e.g. live Linear/Jira/Postgres integration proof, or an unfixable critical/high).
- **Thrash** — same defect regresses 3× OR a Reviewer block recurs unresolved after 3 revision rounds.
- **Drift** — required work falls outside the enumerated scope above.
